# Server — Agent Instructions

## Tech Stack

- **Language:** Python 3.13
- **Framework:** Flask (served via Gunicorn)
- **Database:** MariaDB 11 (via SQLAlchemy 2 ORM)
- **Cache:** Redis 7
- **Container:** Docker (slim Python base image)
- **Frontend:** Static HTML/CSS/JS served by Flask (no build step, no framework)

## Architecture

```
[Android App] --HTTP--> [Flask API] ---> [Redis (live state cache)]
                                    \--> [MariaDB (event history, devices, firmware)]
                                    \--> [Static site (dashboard)]
```

- **Redis** holds real-time vape state (who's vaping right now). TTL-based expiry for device timeout.
- **MariaDB** stores persistent data: device registry, vape event history, firmware metadata.
- **Static site** polls the API and displays current status with fun animations.

## Module Structure

| File | Responsibility |
|------|---------------|
| `app.py` | Flask application, all API routes, auth, background watchdog |
| `models.py` | SQLAlchemy ORM models (Device, VapeEvent, Firmware) |
| `generate_qr.py` | QR code generation utility |
| `site/` | Static frontend (HTML, CSS, JS, GIFs) |
| `Dockerfile` | Container build definition |

## Coding Standards

- **Python best practices:** Type hints for function signatures. Use `pathlib.Path` over `os.path`. F-strings for formatting.
- **Flask patterns:** Use `jsonify()` for API responses. Return appropriate HTTP status codes. Use decorators for auth.
- **SQLAlchemy 2 style:** Use `select()`, `insert()`, `update()` statements — not legacy Query API.
- **Environment config:** All configuration via environment variables with sensible defaults. Never hardcode secrets.
- **Error handling:** Return JSON error responses with meaningful messages. Log exceptions. Don't expose stack traces to clients.

## API Authentication

- Device endpoints require `Authorization: Bearer <VAPE_API_TOKEN>` header.
- Admin endpoints use session-based auth (password login).
- The API token is a shared secret between the Android app and server.

## Key Endpoints

- `POST /vape-update` — Receive vape events from Android app
- `GET /status` — Current vape state (polled by dashboard)
- `GET /firmware/latest` — Latest firmware metadata
- `POST /firmware/upload` — Upload new firmware binary (admin)
- `GET /admin/*` — Admin panel routes

## Frontend (Static Site)

- Vanilla JS — no frameworks, no build tools.
- Polls `/status` endpoint every few seconds.
- Displays fun GIFs/animations when someone is vaping.
- Keep it lightweight and silly — this is a joke project's frontend.

## Docker & Deployment

- Single-stage Dockerfile, Python slim base.
- Gunicorn with 2 workers (sufficient for this scale).
- Docker Compose orchestrates server + Redis + MariaDB.
- Firmware binaries stored in a Docker volume (`/firmware`).

## Best Practices

- Validate all incoming request data (types, lengths, allowed values).
- Use parameterized queries (SQLAlchemy handles this) — never interpolate user input into SQL.
- Keep the API surface small. One endpoint per responsibility.
- Redis keys should be namespaced (prefixed with `WhoIsVaping:`).
- Background threads (watchdog) must handle exceptions gracefully — a crash shouldn't take down the server.
- The frontend should degrade gracefully if the API is unreachable.
