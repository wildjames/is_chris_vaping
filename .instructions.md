# IsChrisVaping — Project Instructions

## Overview

This is a novelty IoT project that detects when someone is using a vape device and displays status on a web dashboard. The system consists of:

- **Firmware** (C++/Arduino on nRF52840): Detects coil activity via GPIO, reports state over BLE.
- **Android App** (Kotlin): Connects to firmware over BLE, relays events to the server via HTTP.
- **Server** (Python/Flask): Receives events, caches state in Redis, persists to MariaDB, serves a web dashboard.
- **CI/CD** (GitHub Actions): Builds Docker images, compiles firmware, enforces PR conventions.

## General Principles

- **Simplicity first.** This is a fun personal project. Avoid over-engineering. Prefer straightforward implementations over clever abstractions.
- **DRY.** Extract shared constants/logic only when there are 3+ usages. Don't pre-emptively abstract.
- **Safety & robustness.** Validate at system boundaries (API endpoints, BLE message parsing, user input). Internal helper functions can trust their callers.
- **No dead code.** Remove unused imports, variables, and functions. Don't leave commented-out code.
- **Consistent naming.** Use the domain language: "device", "coil", "vape event", "ripped/not ripped".

## Architecture Boundaries

```
[nRF52840 Firmware] --BLE--> [Android App] --HTTP--> [Flask Server] --Redis/MariaDB--> [Web Dashboard]
```

- Firmware ↔ App: BLE characteristic messages (`COIL_A:STARTED`, `COIL_A:STOPPED`, etc.)
- App ↔ Server: HTTP POST to `/vape-update` with auth token
- Server ↔ Dashboard: Static HTML/JS polling the server API

## Versioning

- Firmware uses semantic versioning (MAJOR.MINOR.PATCH).
- PR titles touching `firmware/` must be prefixed with `MAJOR`, `MINOR`, or `PATCH`.
- The CI pipeline auto-bumps the version based on the merge commit prefix.

## Environment & Secrets

- All secrets are passed via environment variables — never hardcode tokens, passwords, or URLs.
- Docker Compose uses a `.env` file locally.
- GitHub Actions uses repository secrets.

## Code Review Checklist

1. Does the change follow the single-responsibility principle?
2. Are system-boundary inputs validated and sanitized?
3. Are error cases handled gracefully (no crashes, sensible fallbacks)?
4. Is the change tested where applicable?
5. Does the PR title follow the versioning convention (if firmware is touched)?
